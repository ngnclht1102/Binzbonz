# Task 11 — Comments CRUD Module + Mention Parser

## Type: Backend

## Description
Build the Comments module with CRUD for task comments and project comments, plus an @mention parser that creates wake events.

## Endpoints

### Task Comments
| Method | Path | Description |
|--------|------|-------------|
| GET | `/tasks/:taskId/comments` | List comments for a task |
| POST | `/tasks/:taskId/comments` | Create comment on a task |

### Project Comments
| Method | Path | Description |
|--------|------|-------------|
| GET | `/projects/:projectId/comments` | List project-level comments |
| POST | `/projects/:projectId/comments` | Create project-level comment |

## Mention Parser Logic
On every comment insert:
1. Parse `@name` patterns from `body` using regex (`/@([\w-]+)/g`)
2. Look up matching actors where `type = 'agent'`
3. For each matched agent, insert a `wake_event` with:
   - `agent_id` = matched actor
   - `project_id` = task's project (resolved through hierarchy)
   - `triggered_by` = `'mention'`
   - `comment_id` = the new comment's ID
   - `status` = `'pending'`

## Notes
- `comment_type` is required: `update` | `block` | `question` | `review_request` | `handoff` | `memory_update`
- `actor_id` is required (who posted the comment)
- Comments are append-only — no update or delete
- Mention parser only triggers for `type = 'agent'` actors, not humans

## Acceptance Criteria
- [ ] `POST /tasks/:taskId/comments` creates a comment and returns it
- [ ] Comment with `@dev-1` in body creates a `wake_event` for dev-1
- [ ] Comment with `@brian` (human) does NOT create a wake event
- [ ] Comment with multiple mentions (`@dev-1 @dev-2`) creates multiple wake events
- [ ] `GET /tasks/:taskId/comments` returns comments ordered by `created_at`
- [ ] Project comments work the same way
